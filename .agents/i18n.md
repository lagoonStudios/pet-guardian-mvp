# Internationalization (i18n) Guidance

Use this guidance for all user-facing text and backend-to-frontend messages.

## Core policy

- The app is multilingual by default.
- Every text visible to users must be implemented through i18n.
- Hardcoded user-facing strings in UI components or screens are not allowed.

## Frontend implementation

- Store translations in JSON locale files (for example by language and feature/domain).
- Use translation keys in UI code and resolve them through the i18n layer.
- Keep copy in locale JSON files, not inline in components.
- New UI text must include entries for all supported languages before merge.

## Backend message strategy

- Backend responses should avoid hardcoded, final user-facing text whenever possible.
- Prefer returning stable message codes/keys (and optional metadata) that the frontend translates.
- If backend-originated messages are required, map/parse them into frontend i18n keys before display.
- Keep backend message codes documented and version-safe to avoid frontend drift.

## Consistency rules

- Reuse existing translation keys when semantics match.
- Use consistent key naming conventions by domain/feature.
- Treat message keys as contracts between backend and frontend.
